# Review of the original hardware brief vs. the interrogator repo + 2026 tooling reality

**Status: v0.1 — companion to [`../PLAN.md`](../PLAN.md).** This is the requested audit: what the draft brief already had right, what it missed from the `interrogator` repo, and what July-2026 research corrects. Date: 2026-07-10.

## 1. What the draft got right (kept as-is)

- **Repo split + contract boundary.** Hardware in `interrogator-hardware`, registry stays upstream and is consumed, never forked. This matches ADR-0013 exactly — the USR was *designed with PCB as a named consumer* (`facets.electrical` / `facets.mechanical` are read-only binds for "pcb"; the registry docs literally list PCB as one of the three consumer streams).
- **AssetPin / artifact discipline.** Checksummed, versioned, rev-pinned heavy assets kept out of the software repo — the repo already implements the schema (`assets: {asset_id, rev, object_key, sha256, retrieved}` in every `shared/schemas/sensors/*.yaml`), currently all `null`/TODO. The hardware build is what will finally populate it.
- **Co-design philosophy** (electrical/mechanical/thermal negotiated together, PCB envelope → chassis), the elastic-boundary idea (grow layers before XY), MCAD STEP handoff, the Design Rationale Log, and the human-in-the-loop role shift — all sound and retained.
- **The research mandate itself.** The brief explicitly asked to validate Freerouting HDI capability, mixelpixx MCP reliability, and LLM datasheet parsing before committing. Good instinct — all three came back with material corrections (§3).

## 2. What the draft missed from the interrogator repo

> **Reading note (owner direction, 2026-07-10):** everything below is *evidence and overlap-awareness*, not binding precedent. The build is requirements-first (PLAN §0.1); only the contract artifacts (USR facets, wire/command contract, the #7 integration checkpoint) bind. Prototype implementation details inform, and are otherwise free to be discarded.

1. **The PCB gate already has a definition of done — issue #7.** Exit KPIs: e2e re-validated on PCB, binary IDL (ADR-0004) live with text debug mode, diagnostics re-run vs breadboard golden baseline with no regression, PLAN §7.10 latency budget met. The plan's phases now terminate on these KPIs instead of a generic "production-ready package."
2. **The electrical architecture was already researched upstream — PLAN §7.5.3.18.** Parallel buses with concurrency per bus, VL53L8CX+BNO085 → SPI+DMA, INT-driven acquisition with FIFO watermarks, GPS-PPS input-capture timestamping, mux off the critical path, I3C deferred to v2, MCU candidate list. The draft's pipeline focused on EDA mechanics and skipped that the *electrical* answer largely exists; the plan adopts §7.5.3.18 wholesale.
3. **A device profile view (#123) is the per-instance placement contract** (which sensor on which bus at which address) — distinct from the per-type records. The v1 board must *instantiate* a device profile, not invent its own table.
4. **The single-writer rule.** The validation console is the sole writer of facet validation state; a PCB-side finding (e.g. wrong address) is routed to the facet owner, not patched locally. The draft's "agents fix things autonomously" needed scoping to *this repo's* artifacts only.
5. **Electrical facts are knowingly inconsistent upstream** (inventory CSV vs `board_a_sensor_integration.md` vs `contract.py`; AMG8833 listed on two mux channels; every electrical facet is `pending` awaiting USR-3 SME resolution). Any pipeline that ingests these as ground truth today inherits the contradictions — hence H0's "draft resolution packets, owner ratifies upstream."
6. **`sensor_specs.yaml` bounds are 100 % `confirmed:false`** (conformance all-CONDITIONAL; upstream #27). Diagnostics on the new board can't produce a meaningful no-regression verdict until bounds are ratified — a cross-repo dependency the draft didn't surface.
7. **The bench deviation log is a design-rule goldmine** the draft never referenced: XIAO A/D-pin aliasing, the 0x57 BMV080/MAX30102 collision, VL53L8CX interface-select strap (silent NACK), spectral-chain 100 kHz rise-time ceiling from stacked pull-ups, SGP41 DOA wedging the whole bus (→ per-segment power gating), MAX30102 1.8 V pull-up trap, wrong-library-but-init-passes, 15.77 mm row-spacing caliper catch. All folded into PLAN §2.5.
8. **Thermal/EMI zoning rules already exist and are binding** (PLAN §6.1 zones, §13 mitigation matrix, "Rule of Zero Drift", Geiger/GPS keepout distances, thermal chimney) — the draft's generic "take heat into account" is replaced by these concrete rules.
9. **The sensory-gap register (capability-coverage.md, CAP-4/#65) is the intended input queue for board revisions** — ROI-ranked hardware backlog (GAP-01 AC E-field, GAP-02 selective gas, electrochemical NO for FeNO…). The draft's "new sensor hits the market → we ship it" now has its concrete mechanism.
10. **Regulatory posture is already decided**: `body-physiological` is wellness-only; ship-gate #10 requires per-modality privacy/regulatory packs (SEM-5/#54). Matches the brief's intent; the plan cites the actual anchors. Hardware-specific additions: BMV080 laser class 1, UV-LED eye safety, skin-contact materials.
11. **Deprecated docs trap:** `archive/legacy_docs/{master_topology,fabrication_netlist}.md` conflict with the live pinmap (old netlist, VL53L1X, different UART pins, 5 V on column J…). Flagged so no ingestion agent ever consumes them.
12. **Existing constraints the draft restated loosely, now bound to numbers:** power budget (~400 mA passive / ~950 mA active / ~5 mA sentinel; battery lockout 3.4 V; ~5 h on 2000 mAh vs the 8 h requirement — a real tension made explicit as W-PWR), stream budget (25–30 KB/s continuous, ~100 KB/s peak), sampling-rate floors (BME688 heater cycle, VL53L8CX 15 Hz@8×8, AMG8833 10 fps), transport roadmap (USB-CDC → BLE/WiFi hybrid), and the firmware rule "custom PCB triggers the ESP-IDF move."
13. **Watchdog is disabled (G6) and no INT lines were ever wired** on the breadboard — the INT-driven PCB architecture the draft assumed available actually requires new wiring + firmware work, tracked as a cross-repo dependency.

## 3. Corrections from July-2026 tooling research

1. **HDI blind/buried microvias: drop the assumption.** JLCPCB standard production does **not** offer blind/buried vias (their own Q&A page carries the question unanswered; third-party capability summaries are explicit). What it offers: free resin-filled copper-capped **via-in-pad (POFV) on all 6–20 layer boards**, 0.2 mm min drill, 3.5 mil trace/space on 6+ layers. Our densest parts don't need true HDI — **6–8 layer through-via + POFV** is the design class. If a future revision truly forces 1+N+1 microvias, that's a fab change (PCBWay etc.), not a default.
2. **Freerouting reality** (v2.2.4, May 2026, actively maintained — maintainer is Andras **Fuchs**, not Bodor): fully headless CLI (`-de/-do`, `-inc` to protect nets, `-drc` KiCad-JSON report) + self-hostable REST API (beta); **no diff-pair routing (open FRs #133/#358, maintainer has no bandwidth), no length matching, no copper-pour awareness, no evidence of blind/buried via support anywhere in docs/settings/issues**; and — important — a **documented v2.x routing-quality regression vs v1.9** (discussion #508: v2.x left ~50/300 connections unrouted where v1.9 finished; maintainer acknowledged the headless/API refactor hurt the v1.9 heuristics; the spring-2026 2.2.x releases are the ongoing remediation, incl. infinite-loop/stagnation guards added as late as May 2026). Verdict: grunt-net router behind `-inc`-protected hand routing, benchmarked against v1.9.0 on our own board before trusting v2.x. The draft's "pre-condition net classes and pre-route BGA escapes, then global auto_route" survives only in that reduced role.
3. **mixelpixx/KiCAD-MCP-Server: unverified in practice.** ~1.2k stars; README claims Freerouting integration + JLCPCB export, requires KiCad 9+; but **zero independent end-to-end reports on real boards** were found, and the repo self-describes as AI-generated with no warranty. The draft's "crucial optimization breakthrough… eliminates Quilter for $0" is **not supported by evidence** — treat as an experiment, not infrastructure. Better-engineered alternative to evaluate: **rjwalters/kicad-tools** (v0.13.0, Apr 2026 — format-preserving S-expression round-trip, pure-Python DRC with JLCPCB presets, A* router, bundled MCP server). lamaalrajih/kicad-mcp is solid but read-only (good "eyes", no "hands").
4. **KiCad 10 platform facts:** released 2026-03-20, point releases through **10.0.4 (Jun 2026)**. The official IPC API (`kicad-python`/kipy 0.7.1) is **PCB-editor-only and requires a running GUI in KiCad 9 and 10** (confirmed on the KiCad devlist, Jan 2026: no eeschema API and no headless mode until v11's `kicad-cli api-server`; SWIG bindings removed in v11). Therefore the headless closed loop = **kicad-cli** — `sch erc` / `pcb drc` with `--format json` + `--exit-code-violations` (exit 5 on violations), plus **jobsets** (`kicad-cli jobset run`) for whole output pipelines — augmented by S-expression libraries (kicad-sch-api for schematics, rjwalters/kicad-tools, SKiDL v2.2.3). There is currently **no official schematic automation API at all** — a hard constraint on "AI writes the schematic" ambitions.
5. **LLM datasheet extraction: no benchmark exists** for pinout/footprint extraction accuracy (searched; nearest is generic document-extraction work). Nobody has published evidence it's reliable at 300-page/dense-pin-table scale. 2026 best practice: **verified sources first** — easyeda2kicad.py v1.0.1 (LCSC catalog → KiCad symbol/footprint/STEP; ideal for a JLCPCB-assembled build), SnapMagic Search, Ultra Librarian, SamacSys; AI generation (ProtoFlow-class) only for parts no library has, always human-verified. LLMs are used as *second readers* (cross-checking netlists against datasheets) — which they are good at. This maps cleanly onto the repo's E0→E1→E2 evidence tiers: machine-fetched = E0, human overlay-checked = E1, bench-validated = E2.
6. **Commercial candidates worth a free look, none load-bearing:** Quilter (native KiCad output, free program, credible independent praise on a complex SOM layout), DeepPCB (30-min free trial), atopile (~3.4k stars, active, KiCad layout target — the best fit for the *regeneration* goal but pre-1.0), tscircuit (beta), JITX (enterprise), Diode ($10k–50k/design — out of scope), Cadence/Synopsys AI (enterprise licensing — out of scope). Flux.ai drew sharply negative independent review — avoid.
7. **Overall verdict for this project:** the closed-loop pipeline is real and buildable *today* for: contract ingest → verified library assembly → ERC/DRC CI loops → fab packaging → STEP/MCAD sync. It is **not** real today for: autonomous dense placement, SI-aware routing, unverified LLM footprint generation, or one-shot "ship-ready" regeneration. The plan (§6.2) draws exactly this trust boundary, which also honors the brief's own requirement that a senior engineer's role shift to architectural validation — there is still a human at every promotion gate.

## 4. Requirement tensions surfaced (owner attention needed)

| Tension | Where it bites |
|---|---|
| **8 h ambient battery (R3) vs ~5 h current estimate** | W-PWR budget table at H1 decides: bigger cell (size/weight), leaner ambient set, or relaxed target |
| **AirPods-case north star (R4) vs 16 sensors + apertures + fan + battery** | v1 target stays 120×70×35 mm (repo §6.2); miniaturization is v2+; be explicit that v1 ≠ north star |
| **"Nothing exposed" aesthetic vs sensor physics** | UV-C needs a real aperture (glass absorbs it); BMV080 needs free air; PPG needs optical contact; GNSS wants sky. The aperture-plate design (§5.6–5.7) is the compromise mechanism |
| **"All solid-state, 3 y" vs fan/pumps/valve** | Actuators are wear parts — mounted on a serviceable path, excluded from the sensor reliability claim |
| **15→100+ sensors** | I²C addressing + bus count + power ceiling all cap out well before 100 on one board; the honest scaling story is I3C (v2) + tiered acquisition + possibly satellite boards — flagged for the v2 architecture study |
